## 1. 文档说明

- 文档名称：在线视频与直播网站软件开发计划书
- 版本：v1.0
- 状态：计划阶段提交稿
- 适用阶段：课程大作业计划阶段
- 编制依据：课程大作业要求、在线视频与直播网站选题说明、现有项目文档结构

## 2. 项目背景与编制目的

### 2.1 项目背景

本项目选题为“在线视频与直播网站”。系统面向内容创作者（UP主/主播）和普通用户（观众），目标是在统一平台内完成视频内容的上传发布、分发浏览、实时弹幕互动、轻量级直播以及后台管控等核心业务链路。课程要求最终交付的不只是可运行系统，还包括能够连续反映分析、设计、实现和验收过程的正式文档，因此项目计划必须同时约束系统范围和文档边界。

### 2.2 编制目的

本计划书用于解决三个问题：

- 明确本期必须完成的能力范围，避免在需求分析和实现阶段持续扩张目标。
- 给后续需求、设计、测试、部署和用户手册提供统一口径，确保各文档围绕同一条“视频发布与消费”主链路展开。
- 建立阶段出口、质量基线和风险应对规则，使后续工作可以按“是否满足条件”判断，而不是按口头约定推进。

## 3. 项目目标

### 3.1 交付目标

本项目的交付目标是形成一个可演示、可部署、可答辩的在线视频与直播平台。系统需围绕视频社区的主链路完成闭环，保证创作者能够顺利发布内容，用户能够流畅观看并进行弹幕互动，管理员能够进行有效管控。

为支撑最终展示，交付版本至少应满足以下要求：

- 创作者、普通用户和管理员能够进入各自可见的功能入口。
- 创作者能够完成视频文件的上传、封面设置及元数据编辑。
- 普通用户能够在列表或搜索中发现视频，并进入播放页流畅观看。
- 用户能够在视频播放过程中或直播间内发送和接收实时弹幕，并完成评论点赞互动。
- 管理员能够对违规视频内容和弹幕进行审核与屏蔽。
- 系统能够在演示环境中稳定运行，并支持按固定脚本完成演示。

### 3.2 文档目标

本计划书不仅约束交付范围，还约束后续文档应回答的问题。各类正式文档至少需要继承本计划书中的如下内容：

| 后续文档 | 需要承接的内容 |
| --- | --- |
| 软件需求规格说明书 | 角色边界、核心需求清单、用户场景、验收口径 |
| 软件概要设计说明书 | 模块边界、系统主流程、接口总览、数据流向 |
| 软件详细设计说明书 | 关键对象状态、接口规则（如WebSocket通信机制）、异常处理、页面与数据细节 |
| 测试报告 | 核心需求对应的测试项、主链路回归结果、演示流程验证结果 |
| 部署文档 | 演示环境所需组件、部署步骤、运行检查项 |
| 用户手册 | 创作者端、用户观看端和管理员端的典型操作路径 |

后续文档可以增加细节，但不得偏离本计划书确定的基础范围、边界和阶段出口。

## 4. 项目范围与边界

### 4.1 基础交付范围

本期基础交付只围绕视频内容流转与互动主链路展开。以下能力属于正式承诺范围，后续需求、设计、实现和测试都必须覆盖。

| 能力 | 本期范围 | 最低完成标准 |
| --- | --- | --- |
| 登录与鉴权 | 支持用户注册、登录和退出；基于身份（普通用户/创作者/管理员）限制页面与接口访问 | 各类用户能够进入各自功能入口，未授权请求会被网关拦截 |
| 视频点播(VOD) | 创作者上传视频文件与封面，填写标题标签；用户浏览、搜索视频 | 视频能成功保存并生成可访问链接，播放器能正常加载并拖拽进度 |
| 互动与实时弹幕 | 视频实时弹幕的发送与拉取、视频下方评论区、基础点赞/收藏 | 弹幕能在指定时间点准确渲染；互动数据能正确落库并展示 |
| 轻量级直播 | 创作者开启直播间；用户进入直播间发送实时弹幕；在线人数统计 | 直播间状态流转正常，保证直播间内 WebSocket 弹幕收发正常 |
| 后台管控 | 平台管理员对视频内容的审核管理、对违规弹幕的屏蔽管理 | 违规视频或弹幕被处理后，在前端用户侧不可见 |
| 个人主页与关系 | 用户基本信息展示，关注其他用户与粉丝列表查看 | 能完成“关注/取消关注”流转，并在主页正确展示数量 |

### 4.2 增强与加分候选

下列内容不是基础交付承诺，只能在视频与弹幕主链路稳定、演示流程完整、正式文档已形成闭环的前提下纳入：

| 候选项 | 可纳入的前提 | 纳入后的定位 |
| --- | --- | --- |
| 复杂推荐算法 | 视频主链路流转已稳定，且有基础的按时间排序托底 | 替换首页基础信息流，不影响视频核心播放 |
| 管理员仪表盘 | 基础业务数据已稳定落库，系统性能指标可采集 | 作为大作业加分项中的“系统可观测性”展示 |
| AI Agent 辅助 | 基础视频播放与业务逻辑正确可演示，且新增调用不阻塞主流程 | 作为基于大模型的“AI课代表(视频摘要) / 智能标签提取”扩展功能 |
| Skills 沉淀 | 已实际使用大模型参与文档生成或辅助编码工作 | 作为大模型辅助开发的补充说明材料 |
| UI/UX 美观设计 | 不改变已有业务流程和前后端交互接口规则 | 作为界面统一性和展示效果提升项 |

增强项一旦决定纳入，必须同步补齐对应需求说明、设计说明、测试证据和演示路径；如果无法补齐，仍按“未纳入正式范围”处理。

### 4.3 明确排除范围

以下内容不纳入本期正式交付范围：

- 真实支付接口对接、复杂的虚拟币充值与公会打赏结算系统
- 面向百万级大规模用户的高并发容量测试与底层硬解码优化
- 完整的 Android/iOS 原生客户端同步上线能力
- 与视频播放及弹幕互动主链路无直接关系的功能堆叠

## 5. 开发原则与推进策略

### 5.1 开发原则

- 主链路优先：任何新增内容（如AI或仪表盘）只有在不影响“视频上传发布到观看互动”闭环的前提下才可进入实现范围。
- 文档同步：范围、流程、接口规则或数据表发生变化时，相关文档必须同步更新，禁止长期保留与实现不一致的描述。
- 需求可追溯：每个核心能力都必须同时落实到页面入口、接口支撑、数据落点、测试证据和答辩演示路径。
- 最小可行实现：复杂能力（如直播推拉流）先完成能够支撑核心交互和演示的最小方案（如伪视频流循环+真实WebSocket弹幕），不预先死磕超出当前阶段的底层协议。
- 稳定性优先：加分项、展示项和优化项不得挤占基础主链路的 Bug 修复、前后端联调和验收准备时间。
- 演示可复现：交付结果必须能够通过固定环境、固定测试数据和固定操作路径重复演示，不能依赖临场解释补足缺失能力。

### 5.2 推进策略

项目推进采用“先收口范围，再冻结需求，再形成设计，再打通实现，最后准备验收”的顺序。各阶段输出必须能够直接成为下一阶段输入，避免出现计划书无法支撑需求、需求无法支撑设计、设计无法支撑实现的问题。

在实现阶段，优先保证以下链路完整：

1. 用户成功注册并登录。
2. 创作者成功上传视频文件并填写元数据。
3. 普通用户在首页看到该视频，点击进入并流畅加载播放。
4. 普通用户在播放时发送弹幕，同房间其他用户能实时接收。
5. 管理员在后台能查看到该视频和弹幕的数据记录并进行操作。

只要上述链路中任一关键节点尚不稳定，增强项（如大模型接入、运维监控大屏）均不进入正式实现范围。

## 7. 质量保证与配置管理

### 7.1 质量保证要求

项目质量控制以“能否证明核心需求已落地”为判断标准。至少执行以下检查规则：

- 需求追踪检查：每项核心需求（如发送弹幕）都必须能对应到前端组件入口、后端API支撑、数据库记录、测试通过项和演示步骤。
- 变更一致性检查：范围、接口通信协议、数据结构结构发生变化时，设计文档必须同步修订，不能只改代码不改文档。
- 主链路回归检查：涉及大文件上传、视频状态更新、WebSocket 连接机制的变更，合入代码主干前必须重新检查播放与互动主链路是否被破坏。
- 演示准备检查：演示账号、预置视频资源、演示环境和演示步骤必须提前验证，避免答辩展示时依赖人工补解释。
- 验收材料检查：测试报告通过截图、部署文档命令和用户手册说明必须能在最终提交的版本中找到绝对的对应依据。

### 7.2 配置管理要求

- 分支规范：稳定版本保留在 `main`，集成开发保留在 `dev`，新增特性使用 `feature/*`，缺陷修复使用 `fix/*`。
- 提交规范：提交信息采用 `type: summary` 格式，要求能直接说明本次修改涉及的模块与内容。
- 版本标记：能够形成稳定演示或最终正式交付的版本应打标签（Tag）保存，以便环境重置与回溯。
- 文档管理：正式交付文档统一放置在代码仓库指定目录中；架构或接口结构变更时需同步提交更新后的 Markdown 文档。
- 变更记录：影响系统级范围、接口契约、业务流程或验收口径的修改，应在对应文档开头留下可追踪的版本更新记录。

## 8. 风险管理

| 风险 | 触发情形 | 应对策略 |
| --- | --- | --- |
| 核心需求边界不清导致返工 | 需求分析时过多讨论边缘社交玩法，或不同文档对基础范围（如是否要全量重写流媒体协议）理解不一致 | 严格以本计划书的基础交付范围（4.1）为准收口；新增内容先判断是否属于增强候选，不达标坚决不纳入手册与设计 |
| 大文件及高频网络IO导致系统崩溃 | 视频上传阻塞业务主线程，或 WebSocket 并发连接数过高导致内存溢出 | 设计阶段引入分片上传机制，并利用 Redis 队列为弹幕高频写入做削峰处理；先保证主链路不崩，再追求极致性能 |
| 文档与代码实现脱节 | 接口参数、数据库字段或校验规则已经变化，但设计文档和测试文档未同步更新 | 将文档同步作为变更的必要动作；凡是影响核心交互链路的修改，强制实行“先改文档，后改代码”的原则 |
| 演示环境不可复现 | 演示环境与本地开发环境网络及组件差异过大，部署步骤不完整 | 提前采用 Docker Compose 固化环境组件，录制备用演示视频，并通过部署文档由非开发人员验证从零启动流程 |
| 过早投入加分项挤占主线 | 在视频尚无法正常加载播放时，开始耗费大量时间实现 Grafana 监控面板或调试大模型 Prompt | 铁律：以“主链路稳定、文档闭环、演示可复现”作为增强项的唯一准入条件，不满足条件时严禁启动外围拓展编码 |
| 大模型生成内容未经核查 | 团队成员直接复制粘贴大模型生成的代码片段或设计文档，存在“幻觉”（如调用了不存在的库） | 对大模型产出严格执行人工 Review 复核，只保留通过本地环境运行测试的代码，并按要求单独维护《大模型使用说明》材料 |

## 9. 验收与交付要求

### 9.1 核心需求验收要求

核心需求的验收以“是否形成可证明的闭环”为标准。每项视频业务相关的核心需求至少应具备以下证据：

- 一个明确的用户操作场景（如：用户点开视频并发送一条带颜色的弹幕）
- 一个可交互的页面前端组件入口
- 一组支撑该动作状态流转的 API 接口
- 一份真实的数据库落盘记录或缓存变化
- 一项对应的测试通过结论记录
- 一段清晰的、可用于答辩展示的操作路径

### 9.2 系统交付要求

- 系统必须支撑创作者和普通用户走完全部主链路流转，坚决不能只交付静态 UI 页面或孤立的微服务。
- 交付版本必须能够在演示环境中稳定运行，关键数据展示与演示材料上的截图保持一致。
- 系统应支持演示场景下的并发交互（如多个测试账号在同一视频下发弹幕），不出现明显的连接中断或阻塞性异常。

### 9.3 文档交付要求

- 计划书、需求说明、设计说明、测试报告、部署文档和用户手册之间必须保持强关联的递进关系，前述模型和术语必须为后续文档提供直接依据。
- 文档内容必须反映系统的真实状态，不能在文档中描述无法落地或因时间不够而舍弃的虚假能力。
- 每份正式文档都必须紧扣本计划书定义的范围、边界和验收口径展开，不允许单独在需求或设计文档中私自扩展不受计划约束的交付范围。

### 9.4 大模型使用说明要求

依据课程要求，如果在过程文档撰写、核心业务设计分析、编码或测试脚本生成中使用了大语言模型及相关工具，应单独建立独立的 Markdown 文件予以说明。其中需详细记录大模型的介入场景、人工审查与复核的方式，并附带相关的 Session 对话记录或 Skill 提示词等物料。本计划书只约束该材料需要作为交付物独立存在，不在正文中展开大模型的具体使用细节。
